Stack & arquitectura

Clean Architecture en una sola dirección. La UI nunca habla con HTTP directo.

Mobile MOBILE

view (Widget → Controller) → application (UseCase) → data (Repository → Service) → domain
  • Riverpod estado · go_router navegación · Dio HTTP · either_dart errores (Either<Exception,T>)
  • UseCase: única lógica de negocio, envuelve todo en handlerApiExceptions (nunca lanza).
  • Controller (StateNotifier.autoDispose): loading → fold → success/error.
  • Service: único que usa Dio (dioHttpClientProvider inyecta token, app-source, base URL).
  • Features no se importan entre sí su data/ o application/; lo común va en lib/modules/.

Backend BACKEND

  • Django + DRF · auth por Token (DRF authtoken) · correo por Resend (django-anymail) · Celery + Redis.
  • App de usuarios en core/auth/ (app label users) — modelo User custom (AbstractBaseUser).
  • Todo endpoint mobile valida el header app-source ∈ ("zafira-app","zafira-web").

Login AMBOS

  1. MOBILE — usuario ingresa usuario + contraseña → AuthController.getToken() → UseCase → Service.
  2. BACKEND — POST /api/v1/auth/token/ (CustomAuthTokenApiView) devuelve el token.
  3. MOBILE — guarda token en secure storage; LoginAuthMixin navega a HomeScreen.

Archivos MOBILE

  • UI: view/widgets/login/ → login_screen, login_body, login_header, login_form, login_social_section, login_auth_mixin.
  • Lógica: view/controller/auth_controller.dart, application/{token,server}_usecase.dart, data/{repositories,services,interfaces}/auth_*.
  • Campo de identidad: usuario (no email) — ícono persona, teclado texto.

Registro AMBOS

  1. MOBILE — form: nombre, usuario, cédula, correo, contraseña (+ indicador de seguridad), confirmar.
  2. BACKEND — POST /api/v1/user/create/ (UserCreateApiView / UserCreateSerializerInput): requiere username, email, dni, password; responde 201 o 400 con errores de campo.
  3. MOBILE — 201 → navega a RegisterSuccessScreen (StatusScreen verde). Error → AppNotification con el mensaje del backend.

Archivos MOBILE

  • UI: view/widgets/register/ → register_screen, register_body, register_header, register_form, password_strength_indicator, register_success_screen.
  • Datos: domain/register_request.dart, data/{interfaces,services,repositories}/register_*, application/register_usecase.dart, view/controller/register_controller.dart, view/state/register_state.dart.

Olvidé / restablecer contraseña AMBOS

Flujo por código de 6 dígitos al correo (sin deep-links nativos). Dos fases.

  1. MOBILE — pantalla "¿Olvidaste tu contraseña?": ingresa correo → requestCode().
  2. BACKEND — POST /api/v1/auth/password-reset/request/: genera código, lo guarda hasheado (email_reset_token, expira 15 min), password_reset_pending=True, envía correo. Responde 200 genérico (no revela si el correo existe).
  3. MOBILE — navega a la pantalla de restablecer (lleva el email por extra).
  4. BACKEND/CORREO — el código llega solo al correo registrado.
  5. MOBILE — ingresa código + nueva contraseña + confirmar → confirmReset().
  6. BACKEND — POST /api/v1/auth/password-reset/confirm/: valida código + expiración (check_password), cambia la clave, pending=False, password_reset_count += 1, last_password_reset_at=now.
  7. MOBILE — éxito → AppNotification + vuelve a login.

Mobile MOBILE

  • UI: view/widgets/{forgot_password,reset_password}/.
  • Datos: domain/password_reset_confirm.dart, data/{interfaces,services,repositories}/password_reset_*, application/password_reset_usecase.dart, view/controller/password_reset_controller.dart, view/state/password_reset_state.dart.

Backend BACKEND

  • core/auth/api/v1/auth/views/password_reset.py (request + confirm).
  • .../serializer/password_reset.py, core/auth/utils/email.py (send_password_reset_code_email), core/auth/tasks.py (Celery).
  • Envío: directo activo (pruebas) + .delay() Celery comentado.
Seguridad: respuesta genérica · código hasheado · expira 15 min · un solo uso · el código solo llega al correo de la BD (jamás a uno aportado por quien solicita).

Conexión (banner online/offline) MOBILE

Portado de hey-support. Detecta en 2 capas y avisa con un banner.

  • connectivity_plus (¿hay red?) + internet_connection_checker_plus haciendo ping a Flavor.server (¿responde el backend?).
  • 3 estados: verde "conectado" · amarillo "modo offline" (red sin backend) · rojo "sin conexión".
  • Provider global; el OnlineWidget se monta una vez en el ShellRoute (no se re-renderiza al navegar).
  • Archivos: lib/modules/connection/ (interface, repository, usecase, controller, state, widgets/online_widget) + core/enum/connection_status.dart.

Cómo correr / pendientes

Correr MOBILE

fvm flutter pub get
fvm flutter run -t lib/main/main_dev.dart --flavor dev
  • Flutter pineado en .fvmrc = 3.41.9.
  • Flavors Android dev/prod en android/app/build.gradle.kts.
  • URL del backend en .env → DEVELOP_ENV (vía envied).

Pendiente / a vigilar

  • BACKEND: correr makemigrations + migrate (campos de reset) y probar el correo.
  • BACKEND: ALLOWED_HOSTS debe incluir la IP del host (ej. 192.168.0.3) y correr en 0.0.0.0:8000.
  • MOBILE: para backend http://, el manifest necesita usesCleartextTraffic="true" (DEV).
  • MOBILE: Home navega solo a placeholder; falta el contenido real por sección.
  • OPCIONAL: deep-link real del correo, rate-limiting del reset, login social (Google/Apple), Firebase init.
